Self-Debugging

AI-Assisted Bug Identification and Fixing

Code Framework 12 min read Intermediate

Debugging with AI Assistance

Self-Debugging is a prompting technique where AI models identify, explain, and fix bugs in code. This approach leverages the model's ability to trace logic, interpret errors, and reason about expected vs actual behavior‚Äîturning debugging from a hunt into a conversation.

Core Insight

Models can often spot bugs they would have created. The fresh perspective of re-examining code for errors, combined with explicit bug-finding instructions, activates different reasoning patterns than initial generation.

Debugging Strategies

Different debugging approaches suit different situations. Choose based on what information you have available.

Error-Driven Debugging

Start with an error message and work backward to find the root cause.

Prompt structure:

"This code produces the following error: [error message]

Here is the code: [code]

Please:

  1. Explain what this error means
  2. Identify the line(s) causing the error
  3. Explain why the error occurs
  4. Provide the corrected code"

Best for: Runtime errors, exceptions, compile errors, type mismatches.

Trace-Based Debugging

Walk through code execution step-by-step to find where behavior diverges from expectations.

Prompt structure:

"Trace through this code with input [input]:

[code]

Show the value of each variable after each line. Expected output is [expected], but actual output is [actual]. Where does the execution diverge from the expected path?"

Best for: Logic errors, wrong output, off-by-one errors, loop issues.

Explanation-Based Debugging

Have the model explain what code does line-by-line. Bugs often become apparent during explanation.

Prompt structure:

"Explain this code line by line, describing what each line does and why:

[code]

The code is supposed to [intended purpose]. As you explain, note any lines that don't align with this purpose."

Best for: Understanding unfamiliar code, subtle logic errors, code you didn't write.

Test-Driven Debugging

Provide failing test cases and have the model fix code to pass them.

Prompt structure:

"This function fails these test cases:

  • Input: [input1] ‚Üí Expected: [expected1], Got: [actual1]
  • Input: [input2] ‚Üí Expected: [expected2], Got: [actual2]

Here is the function: [code]

Fix the function so all test cases pass. Explain what was wrong."

Best for: Clearly defined expected behavior, regression bugs, edge case failures.

The Self-Debugging Workflow

An effective debugging workflow that combines generation and debugging in an iterative loop.

1

Generate Initial Code

Create the first version using standard code prompting techniques.

2

Test the Code

Run the code against test cases or in your environment. Collect any errors or unexpected outputs.

3

Debug with Feedback

Provide the code and the error/failure information to the model. Ask it to identify and explain the bug.

4

Apply Fix

Implement the suggested fix, either by having the model generate corrected code or manually applying the change.

5

Verify and Iterate

Test again. If issues remain, repeat the debug cycle with new error information.

Effective Debug Prompts

Named after rubber duck debugging‚Äîexplaining code to find bugs through the act of explanation.

"I'm going to show you code that has a bug. Please act as my rubber duck: explain what each section of this code does as if teaching a junior developer. The explanation process often reveals the bug."

Ask the model to actively look for problems rather than assume the code is correct.

"Review this code as a skeptical code reviewer looking for bugs. Assume there IS a bug somewhere. Check for: off-by-one errors, null/undefined handling, edge cases, type mismatches, and logic errors. What could go wrong?"

Provide a hypothesis about what might be wrong and ask the model to validate or refute it.

"This code returns the wrong value. I think the bug might be in the sorting logic. Please examine the sorting section specifically, trace through it with example input [1, 3, 2], and tell me if my hypothesis is correct or if the bug is elsewhere."

Request the smallest possible change to fix the bug, avoiding unnecessary refactoring.

"Find and fix the bug in this code. Make the MINIMAL change necessary‚Äîdo not refactor, rename, or restructure anything else. Show me only the lines that need to change and explain why."

Best Practices

Include Full Context

Provide the complete error message, stack trace, and relevant code. Partial information leads to partial solutions.

Specify Expected Behavior

Always state what the code should do. The model needs to know the target to identify the deviation.

Isolate the Problem

If possible, reduce the code to a minimal example that reproduces the bug. Less noise means better analysis.

Verify Fixes

Don't blindly apply suggested fixes. Understand why the fix works and test it against your failing cases.
