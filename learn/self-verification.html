Self-Verification

Validating Answers Through Systematic Checking

Advanced Framework 10 min read Intermediate

Trust But Verify

Self-Verification is a prompting technique that adds a validation step after answer generation. Rather than accepting the model's first response as final, this approach asks the model to check whether its answer actually satisfies the original requirements.

Core Insight

Verification is often easier than generation. A model that produces an incorrect answer can frequently identify the error when explicitly asked to check its workâ€”similar to how humans catch mistakes when proofreading.

How Self-Verification Works

The technique follows a generate-then-verify pattern that separates answer production from answer validation.

1

Generate Answer

The model produces a candidate answer using any appropriate technique (direct response, chain-of-thought, etc.).

2

Formulate Verification

The answer is transformed into checkable conditions or the problem is reformulated to test the answer's validity.

3

Execute Check

The model systematically verifies whether the answer satisfies all requirements and constraints.

4

Accept or Regenerate

If verification passes, the answer is accepted. If it fails, the model generates a new answer informed by the failure.

Verification Strategies

Different verification approaches suit different problem types. Choosing the right strategy maximizes error detection.

Backward Verification

Work backwards from the answer to see if it leads to the original problem conditions. This is particularly powerful for mathematical and logical problems.

Problem: "If 3x + 7 = 22, what is x?"

Answer: x = 5

Verification: "Substitute x = 5 back into the equation: 3(5) + 7 = 15 + 7 = 22 âœ“"

Best for: Math problems, equation solving, logical deductions, puzzles with verifiable solutions.

Constraint Checking

Verify that the answer satisfies all stated and implied constraints from the original problem.

Problem: "Schedule a 2-hour meeting for 5 people next week, avoiding Monday mornings and Friday afternoons."

Sanity Checks

Apply common-sense validation to catch obviously wrong answers, even when formal verification isn't possible.

Problem: "Estimate the number of piano tuners in Chicago."

Answer: 150 piano tuners

Combining Techniques

These techniques can be combined. For example: use Self-Consistency to generate candidates, Self-Verification to filter incorrect ones, and Self-Refine to polish the selected answer.

Implementation Patterns

Include verification instructions in the original prompt, asking the model to check its work before finalizing.

Prompt: "Solve this problem, then verify your answer is correct by checking it satisfies all conditions. If verification fails, solve again."

Pros: Simple, single API call

Cons: Verification may be superficial

Generate answer first, then send a separate verification prompt with the answer and original problem.

Turn 1: "Solve: [problem]"

Turn 2: "Given this problem: [problem] and this proposed answer: [answer], verify whether the answer is correct. Check each constraint."

Pros: More thorough verification, clear separation

Cons: Two API calls, higher latency

If verification fails, regenerate with feedback about what went wrong.

Loop:

  1. Generate answer
  2. Verify answer
  3. If failed: "Your previous answer failed because [reason]. Solve again avoiding this error."
  4. Repeat until verified or max attempts

Pros: Highest accuracy, learns from mistakes

Cons: Variable cost, potential for loops

Best Practices

Make Constraints Explicit

List all requirements clearly so the model knows exactly what to verify. Implicit constraints are often missed during checking.

Request Structured Output

Ask for verification results in a consistent format (e.g., checklist with pass/fail for each constraint) to make failures easy to identify.

Set Maximum Attempts

Always limit regeneration attempts (typically 3-5) to prevent infinite loops when the model cannot find a valid solution.

Include Error Context

When regenerating after a failed verification, include what specifically failed so the model can avoid repeating the same mistake.

When to Use Self-Verification

Ideal For
Math, logic, constraint satisfaction, code, factual questions with verifiable answers
Less Suitable
Creative writing, subjective opinions, open-ended exploration, tasks without clear correctness criteria

High-Value Use Cases

  • Mathematical calculations and word problems
  • Code that must meet specific requirements
  • Data extraction with completeness requirements
  • Scheduling and resource allocation
  • Format compliance (JSON, specific structures)
  • Multi-step reasoning with checkable conclusions

Related Frameworks
